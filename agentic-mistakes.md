## Format d'entrée

```
### [DATE] — [CATÉGORIE] — [Titre court]
- **Contexte** : ce que je faisais / ce qu'on investiguait
- **Erreur** : ce que j'ai dit/fait de faux
- **Cause racine du raisonnement** : pourquoi ce raisonnement était défectueux (biais, hypothèse non vérifiée, donnée partielle, etc.)
- **Comment corrigé** : qui/quoi a remis sur la bonne piste
- **Impact (uniquement si connu)** : coût concret (tours gaspillés, action quasi-engagée, erreur quasi-commit)
- **Lesson** : règle à appliquer dans le futur pour ne pas répéter
```

## Catégories

- **DIAGNOSTIC** : mauvaise conclusion sur la cause d'un bug
- **PROPOSITION SANS DOC** : recommandation d'une étape non vérifiée dans la doc du provider
- **HYPOTHÈSE** : raisonnement basé sur une supposition non testée
- **COMMUNICATION** : phrasing imprécis, omission importante, présentation prescriptive contraire à la doctrine projet
- **EXÉCUTION** : tool/commande mal utilisée (mauvais flag, mauvais path, mauvais ordre)
- **RECHERCHE** : recherche insuffisante avant d'agir (codebase ou web)

## Entrées

### 2026-05-13 — DIAGNOSTIC — Conclusion sur prefix de données réseau, ignorant la transition d'état
- **Contexte** : debug live d'un upload Arclight qui semblait figé à 35% sur la preview Vercel. J'ai utilisé Chrome DevTools MCP pour sniffer le réseau.
- **Erreur** : j'ai listé ~24 polls (tous 200 OK avec `data: null`) et conclu "le bug = `UPSTASH_REDIS_*` manquant → chaque invocation serverless Vercel a son propre `InMemoryStore` → les writes du POST upload ne traversent pas vers les GET polling". J'ai même commencé à proposer de migrer le storage du progress vers Postgres.
- **Cause racine du raisonnement** : tirer une conclusion d'un **préfixe** de série temporelle (`pageSize` du `list_network_requests` plafonné implicitement à ~100 entries), sans s'être assuré que le **steady-state** ou la **transition d'état** soient visibles dans les données. J'ai ignoré que le user m'avait déjà décrit le pattern "200 puis 404 après ~1 min" plus tôt dans la conversation.
- **Comment corrigé** : l'utilisateur m'a poussé back : *"je pense que tu te trompes, je vois des 404 moi"*. J'ai re-listé avec `pageSize=200` et les 404 sont apparus immédiatement à `reqid=112`, avec headers Clerk explicites (`x-clerk-auth-message: JWT is expired`) confirmant le vrai diag.
- **Impact** : ~1 tour gaspillé sur une mauvaise piste. Risque d'avoir commit un refactor inutile (progress vers Postgres) si non corrigé. Le diag Clerk avait déjà été correctement identifié plus tôt dans la session — je m'en suis détourné à cause d'un dump partiel.
- **Lesson** : pour tout bug **time-dependent** (auth expiry, cache eviction, refresh background, race conditions, cold-start vs warm-start), **observer le steady-state OU la transition** AVANT de conclure. Si le user rapporte un symptôme que je ne vois pas encore dans les données, default = *"j'attends / je cherche plus / j'élargis le pageSize"* plutôt que *"je trouve une explication alternative"*. Une hypothèse confirmée plus tôt ne doit pas être abandonnée juste parce que le dump suivant ne montre pas encore le failure.

### 2026-05-13 — PROPOSITION SANS DOC — Recommandation de modifier un réglage Clerk qui n'existe pas
- **Contexte** : après confirmation que le JWT Clerk (TTL 60s) était la cause des 404 sur la preview Vercel, je voulais proposer un quick-win pendant qu'on bumpait le SDK.
- **Erreur** : j'ai recommandé *"va dans Clerk Dashboard → Sessions → 'Session token lifetime' et passe de 60s à 5min"*. - **Cause racine du raisonnement** : j'ai supposé l'existence du réglage en m'appuyant sur ma connaissance générique des produits SaaS d'auth (Auth0, AWS Cognito et autres exposent ce knob). Je n'ai PAS lu/vérifié la doc Clerk avant de recommander une action concrète à l'utilisateur.
- **Comment corrigé** : l'utilisateur a fait l'aller-retour dans le dashboard et m'a confronté à l'absence du toggle via screenshots. J'ai ensuite reconnu la limitation produit et annulé la piste.
- **Impact** : ~2 tours gaspillés à guider l'utilisateur dans des screens inutiles. Érode la confiance — l'utilisateur m'a explicitement reproché de proposer "des choses infaisables" et a demandé l'ajout d'une règle globale pour ne plus jamais le refaire (cette règle existe maintenant dans `~/.claude/CLAUDE.md`).
- **Lesson** : **NE JAMAIS proposer de solution technique impliquant un provider** (Clerk, Vercel, Stripe, OpenAI, etc.) sans avoir d'abord vérifié dans la doc/dashboard que l'option recommandée existe et est accessible au plan de l'utilisateur. Si l'accès à la doc n'est pas immédiat, dire explicitement *"je ne sais pas si c'est exposé, peux-tu vérifier le dashboard X ?"* plutôt qu'affirmer. La connaissance générique d'une catégorie de produit n'est PAS une garantie de l'existence d'une feature spécifique chez un provider donné.

### 2026-05-13 — DIAGNOSTIC — Abandon d'une hypothèse pourtant correcte (deux bugs concurrents non distingués)
- **Contexte** : suite de la session du 2026-05-13 sur l'upload Arclight, après que l'utilisateur ait pointé les 404 que j'avais ratés (cf. entrée précédente).
- **Erreur** : une fois le diag Clerk JWT confirmé, j'ai **abandonné entièrement l'hypothèse Redis/InMemoryStore** que j'avais formulée plus tôt, en écrivant explicitement *"mon hypothèse Redis : à côté du sujet réel"* dans mes annonces et mon entrée `agentic-mistakes.md`. Mauvaise interprétation : j'ai supposé que mon hypothèse initiale était une *alternative* à la vraie cause, alors qu'en réalité **les deux bugs coexistaient** :
  1. Bug Clerk JWT → 404 sur les polls après ~60 s
  2. Bug Redis manquant → `data:null` sur les polls 200 (avant que les 404 ne commencent)
- **Cause racine du raisonnement** : framing binaire "ma théorie était fausse vs juste". Quand l'utilisateur a montré que le diag Clerk était le vrai sujet des 404, j'ai conclu que le diag Redis était nul et non avenu, sans vérifier si les *autres* symptômes (notamment `data:null` sur les 200) restaient inexpliqués.
- **Comment corrigé** : après le push du bump SDK 6.39.3, on a relancé un upload en live ; on a observé que les 404 revenaient ET que tous les polls 200 retournaient `{data:null}`. Cette persistence du `data:null` malgré l'absence de 404 a forcé la prise en compte du bug Redis comme bug indépendant.
- **Impact** : ~1 commit/déploiement supplémentaire (le bump SDK seul, qui n'a même pas suffi pour son bug ciblé). Mémoires et `agentic-mistakes.md` initialement rédigés avec une formulation "j'ai eu tort sur Redis" qu'il faudra nuancer en post-mortem ("j'ai eu tort de croire que Redis était la cause des 404 ; j'ai eu raison sur l'existence du bug Redis").
- **Lesson** : **un bug confirmé n'invalide pas les autres hypothèses encore non testées**. Pour chaque hypothèse écartée, vérifier qu'elle explique l'absence de symptômes *résiduels*. Si un symptôme reste inexpliqué après le diag retenu (ici : `data:null` après les 404), c'est qu'il y a (au moins) un deuxième bug. Annoncer *"hypothèse X était une alternative qui s'est révélée fausse"* est trop fort si X n'a jamais été falsifié, juste mis de côté.

Toujours grep large sur la surface authentifiée (`api/(documents|deals|analyze|...)`) et statuer cas par cas. La spec d'un auditeur indique *au moins* ces fichiers, pas *seulement* ceux-là.

### 2026-05-13 — COMMUNICATION — Réponse 422 annonçait `refundedCredits: 2` sans vérifier le résultat du refund
- **Contexte** : Phase 1.4 du plan Codex (refund retry OCR). J'ai ajouté un `await refundCreditAmount(...).catch(() => undefined); chargedCredits = 0;` dans la branche 422 du retry route, puis renvoyé `{ refundedCredits: 2 }` en dur dans la réponse.
- **Erreur** : la réponse annonçait `refundedCredits: 2` **inconditionnellement**, même si `refundCreditAmount` retournait `{ success: false }` (provider de crédits down, idempotency conflict, DB error transient) ou throwait. Le `.catch(() => undefined)` avalait l'erreur sans la propager. Conséquence : utilisateur reste débité de 2 crédits, l'API lui dit "remboursé". Pire qu'avant : ça donne une fausse confiance.
- **Cause racine du raisonnement** : j'ai conçu le refund comme une étape "best-effort de plus" plutôt que comme une opération financière dont le résultat doit être véridique vis-à-vis du client. Le `.catch(() => undefined)` cargo-culté depuis le catch existant a masqué le fait que `refundCreditAmount` peut retourner `{ success: false }` sans throw.
- **Comment corrigé** : audit Codex pointe la lacune. Refactor : capture du résultat dans `let refundedCredits = 0; let refundFailed = false;`, set conditionnel sur `refund?.success`, `console.error` détaillé en cas d'échec, payload renvoie le vrai état (`refundedCredits: 0, refundFailed: true` si échec). Tests ajoutés pour les deux modes d'échec (return success:false, throw).
- **Impact** : 1 tour de gate à refaire. Pas d'impact prod (Phase 1 pas encore mergée).
- **Lesson** : pour toute opération financière (refund, charge, deduct), **ne jamais swallow le résultat avec `.catch(() => undefined)` ni hardcoder le montant dans la réponse**. Capturer la valeur de retour, vérifier `success` explicitement, traiter `false` ≠ throw comme deux cas distincts, et faire en sorte que la réponse client soit véridique même quand l'opération de crédits a partiellement ou totalement échoué. "Best-effort" n'est pas une excuse pour mentir au client.

### 2026-05-13 — SÉCURITÉ-RAISONNEMENT — Phase 2 a refactoré le cleanup blob sans modéliser la primitive de delete arbitraire
- **Contexte** : Phase 2 du plan Codex. J'ai déclaré le gate vert en pensant avoir résolu (a) cleanup temp blob orphelin, (b) cleanup final blob orphelin, (c) cascade delete deal, (d) check pathname-vs-dealId.
- **Erreur** : J'ai armé `cleanupSourceUpload = uploadInput.cleanupSourceUpload` IMMÉDIATEMENT après readUploadInput, AVANT la vérification d'ownership. Le cleanup s'exécutait via `bailWithCleanup` dès la première early-return. Et `validateTemporaryBlobUrl` ne bindait pas l'URL au pathname. Conséquence : un caller authentifié pouvait submit `{ blobUrl: "<URL d'un autre tenant>", blobPathname: "<my-dealId>/foo.enc" }`. Au premier échec downstream (MIME, CUID invalid, etc.), je supprimais le blob de l'autre tenant. Codex a explicitement nommé ça "primitive de delete arbitraire". J'avais le code sous les yeux et je n'ai pas vu le contre-modèle.
- **Cause racine du raisonnement** : J'ai pensé le cleanup comme "un nettoyage de ressources locales" sans modéliser qu'il opère sur un blob *que je ne possède pas encore prouvément*. La validation `URL.pathname starts with tmp/document-uploads/` me donnait l'illusion de safety alors qu'elle ne ferme aucune des deux failles (URL↔pathname binding ET pathname-vs-dealId). J'aurais dû me poser la question "qu'est-ce que je peux faire à un blob qui ne m'appartient pas" comme grille de threat-modeling AVANT de m'auto-déclarer le gate vert.
- **Comment corrigé** : Audit Codex pointe le P1. J'ajoute (1) binding strict `URL(blobUrl).pathname === '/' + blobPathname` dans validateTemporaryBlobUrl, (2) réordonne le handler pour n'armer le cleanup qu'APRÈS deal.findFirst({ userId }) succès, (3) durcis upload/client pour exiger `pathname.startsWith(tmp/document-uploads/${dealId}/)`. 5 tests route ajoutés couvrant les 4 invariants critiques.
- **Impact** : 1 audit refait. Pas d'impact prod (Phase 2 pas mergée). Mais c'est l'erreur agentique la plus dangereuse jusqu'ici : delete arbitraire authentifié.
- **Lesson** : pour TOUTE opération destructive (`delete*`, `unlink*`, `revoke*`, `truncate*`), avant de l'armer, écrire explicitement : "à ce stade du code, je sais que la cible appartient à l'utilisateur courant parce que <preuve concrète et vérifiable>". Si je ne peux pas remplir cette phrase, je n'ai pas le droit d'armer. La preuve doit être *postérieure* à l'arming — pas se baser sur "le client est honnête" ou "la validation de format suffit". Threat-model : "si le caller ment sur ce paramètre, que peut-il faire avec mon endpoint ?"

### 2026-05-13 — RECHERCHE — Phase 3 mapping incomplet : 3 sites de lecture artifact ratés
- **Contexte** : Phase 3 (Privacy DB). J'ai mappé la surface de lecture `page.artifact` + `page.textPreview` en début de Phase 3.0 et listé 5 sites à wrapper avec `safeDecryptJsonField`. Codex a audité et trouvé 3 sites supplémentaires que j'avais ratés : `extraction-readiness-policy.ts` (toxic gate / readPageVerificationState), `evidence-ledger/index.ts:asRecord`, et `extraction-reuse.ts` (pour le clonage write).
- **Erreur** : Mon grep initial était trop étroit. J'ai cherché `"artifact:|textPreview:"` (sites d'**accès JSON** ou d'**écriture Prisma**) au lieu de `"\.artifact\b"` (toute lecture du champ). Conséquence : j'ai vu les writes (Prisma.create/update avec `artifact: ...`) et les reads via le sélecteur Prisma (`artifact: true`), mais j'ai raté les usages internes `page.artifact` dans des fonctions de policy / ledger / reuse. Chaque site raté est devenu un fail-open Phase 3 distinct :
  - `isPageArtifactToxic(encrypted)` → toujours false → bypass UNVERIFIED_ARTIFACT.
  - `evidence-ledger(encrypted)` → 0 claims structurés → agents perdent l'evidence.
  - `extraction-reuse(legacy source)` → target plaintext sur disk → re-fuite du corpus.
- **Cause racine du raisonnement** : J'ai assimilé "mapping de la surface" à "lister les requêtes Prisma". C'est partiel : ce qui compte c'est tout chemin de code qui lit le champ après extraction de la DB — y compris quand il traverse plusieurs fonctions / modules. La régex grep aurait dû être plus large (`\.artifact\b` AU MINIMUM, idéalement `(artifact|textPreview)`). J'ai pris un raccourci de mapping qui a laissé passer 3 bugs P1 dont un fail-open sur le gate d'analyse.
- **Comment corrigé** : Audit Codex liste les 3 sites. 3 tests RED écrits AVANT fix pour prouver chaque bug. Fix appliqué à chacun (décryption inside policy module, asRecord, et re-encryption au clonage reuse). 5/5 tests GREEN après. Audit transversal final via `grep -rnE "page\.artifact|page\.textPreview"` qui montre que TOUS les sites passent maintenant par `safeDecryptJsonField`/`safeDecrypt` ou par les helpers `reEncryptArtifactForReuse`/`reEncryptTextPreviewForReuse`.
- **Impact** : 1 audit refait. Les 3 bugs auraient eu un impact prod sévère :
  - Toxic gate : bypass du contrôle d'analyse sur tout doc post-Phase-3 → fail-open silencieux.
  - Evidence ledger : qualité d'analyse dégradée silencieusement (claims numériques perdus).
  - Reuse : violation directe du gate audit Codex "corpus brut lisible en DB".
- **Lesson** : pour un sweep de sécurité (chiffrement, auth, ownership) qui touche un champ DB, **le grep d'audit doit être sur le NOM DU CHAMP**, pas sur le pattern d'usage suspecté. `grep "page.artifact"` aurait trouvé les 8 sites au lieu de 5. Et pour les phases de transformation (ici : forme stockée d'un champ change), modéliser explicitement TOUTES les transitions : `legacy → encrypted-read`, `encrypted → encrypted-read`, `legacy → re-encrypted-write` (extraction-reuse). Le 4e cas (re-encrypted-write) m'a été invisible parce que je pensais "envelope-to-envelope copy fonctionne" sans réaliser que la source pouvait NE PAS être un envelope.

### 2026-05-13 — API-DESIGN — safeDecryptJsonField collapsait 3 sémantiques en null → fail-open
- **Contexte** : J'avais conçu `safeDecryptJsonField(value): T | null` pour le cas read-and-display : `null` = "rien à afficher, fallback safe". L'helper agglomère 3 états sémantiquement très différents : (1) input null/undefined (absent), (2) input plaintext null (legacy absent), (3) input envelope chiffrée présente mais indéchiffrable (corrompue).
- **Erreur** : Quand j'ai utilisé cet helper pour câbler le toxic gate Phase 3.5(a) et le reuse Phase 3.5(c), j'ai considéré que "null = no artifact" était la bonne sémantique. Faux pour les chemins security-sensitive : "absent" → continuer normalement, "corrompue" → fail-closed. Codex a refait la repro live : `isPageArtifactToxic({ _enc: "ad1", data: "garbage", v: 1 }) === false`. Le toxic gate était fail-open sur tout envelope tampered.
- **Cause racine du raisonnement** : J'ai exporté un helper "ergonomique" (`T | null`) sans prendre en compte les contraintes des consumers security-sensitive qui ont besoin d'observer chaque état séparément. Pattern classique : une API qui collapse des erreurs en valeur sentinel est commode pour 80% des callers mais incorrecte pour les 20% qui prennent des décisions de sécurité. J'aurais dû dès Phase 3.1 exposer un type discriminé pour les callers stricts, et garder `safeDecryptJsonField` comme wrapper de commodité.
- **Comment corrigé** : Audit Codex pointe le fail-open résiduel. J'ai ajouté `tryDecryptJsonField(value): DecryptedJsonFieldResult<T>` qui retourne `{ kind: "absent" | "plaintext" | "decrypted" | "corrupted", value?, reason? }`. `safeDecryptJsonField` réécrite comme wrapper pour rétrocompat. Toxic gate utilise `tryDecryptJsonField` et fail-closed sur `corrupted`. Extraction-reuse aussi throw sur `corrupted`. 2 tests RED→GREEN (envelope corrompue → toxic, envelope corrompue source → throw au lieu de DbNull).
- **Impact** : 1 audit refait. Si non corrigé : rotation de clé de chiffrement ou tampering DB → fail-open silencieux sur le gate d'analyse. Severity P1.
- **Lesson** : quand on conçoit un helper qui sera utilisé par des security gates, **prévoir dès le départ un type retour qui distingue les états d'erreur**. Une API qui retourne `T | null` est anti-pattern pour les paths où "null" peut signifier (a) "pas de donnée, ok continue" ou (b) "donnée corrompue, fail-closed". Le coût d'un type discriminé en plus est faible ; le coût d'un fail-open silencieux est énorme. Règle : si l'helper est appelé depuis un site qui décide "block ou pas", il doit pouvoir répondre "je sais pas" différemment de "il n'y a rien".

### 2026-05-13 — API-DESIGN — Réutilisé safeDecrypt dans un fail-closed → swallow silencieux, try/catch externe code mort
- **Contexte** : Mon fix Phase 3.5(d) pour `reEncryptTextPreviewForReuse` voulait fail-closed sur un textPreview chiffré indéchiffrable. J'ai écrit `try { const plaintext = safeDecrypt(stored); return encryptText(plaintext); } catch { throw new CorruptedSourceArtifactError(...) }`.
- **Erreur** : J'ai oublié que `safeDecrypt` contient SON PROPRE try/catch qui retourne l'input verbatim sur échec — précisément ce qu'on ne veut pas dans un security gate. Mon try/catch externe ne se déclenchait JAMAIS parce que `safeDecrypt` ne throw jamais. Conséquence : un textPreview base64 corrompu (heuristique `isEncrypted` vraie, AES auth-tag faux) était traité comme plaintext et ré-encrypté tel quel → le ciphertext original corrompu re-apparaît dans la nouvelle row. Repro live Codex : `safeDecryptReturnedOriginal=true, clonedDecryptsToOriginalCiphertext=true`.
- **Cause racine du raisonnement** : J'ai pris pour acquis que `safeDecrypt` throw sur échec (puisque "Decrypt" implique une opération qui peut échouer). Mais l'helper est conçu "safe" = "ne throw jamais, fallback gracieux". Mauvaise lecture du contrat. J'aurais dû lire le code de `safeDecrypt` avant de l'utiliser dans un try/catch. Ironie : la veille j'ai exactement écrit l'entrée précédente "API qui collapse erreurs en sentinel = anti-pattern pour gates", et 5 minutes plus tard j'ai réutilisé `safeDecrypt` dans un gate.
- **Comment corrigé** : Audit Codex pointe le fail-closed buggé. J'ai exposé `tryDecryptText(text): DecryptedTextResult` (miroir exact de `tryDecryptJsonField`). `reEncryptTextPreviewForReuse` réécrit en switch sur `result.kind` : `corrupted` → throw. Test RED→GREEN explicite : ciphertext corrompu (auth-tag flipped) → throw au lieu de re-encrypted-as-plaintext. + 4 tests directs sur `tryDecryptText` qui assert explicitly que `safeDecrypt` et `tryDecryptText` diffèrent sur input corrompu.
- **Impact** : 2 audits refaits sur le même chemin Phase 3 (3.5(c) puis 3.5(e)). Pas d'impact prod (jamais mergé). Mais c'est un signal fort que mon process de fail-closed manque de rigueur : je dois VÉRIFIER que la fonction utilisée throw, pas le supposer.
- **Lesson** : **avant d'envelopper un appel dans un try/catch, lire le code de la fonction appelée et confirmer qu'elle peut throw**. Si elle commence par `try { ... } catch { return input; }`, le try/catch externe est code mort. Pour les opérations security-sensitive, préférer un helper explicitement strict (préfixe `try*` qui retourne un Result discriminé) plutôt qu'un wrapper "safe" suivi d'un try/catch. Et règle pratique : si j'écris une entrée agentic-mistakes sur un anti-pattern aujourd'hui, je dois grep mon propre code pour voir si je viens de réintroduire le même anti-pattern ailleurs.

### 2026-05-14 — VÉRIFICATION — Read tool a rendu des octets NUL comme espaces → j'ai failli rejeter un finding Codex correct
- **Contexte** : Ré-audit Codex du fix-up Phase 4.3. Codex signalait que `acquireDocumentLineageLock` construisait sa clé de lock avec des octets NUL (0x00) réels, qui font crash `hashtext()` côté PostgreSQL.
- **Erreur** : J'ai relu le fichier avec le Read tool, qui a affiché les NUL comme des espaces. J'ai conclu "le code utilise des espaces, pas des NUL — Codex a mal lu" et j'ai commencé à rédiger une réponse pour repousser le finding. C'est seulement par prudence que j'ai lancé `od -c` avant de répondre — qui a montré `d o c - l i n e a g e \0 $ {` : Codex avait raison, c'était bien moi qui avais écrit `\0` dans mon Edit précédent (le tool a interprété l'échappement comme un vrai NUL).
- **Cause racine du raisonnement** : J'ai traité la sortie du Read tool comme une représentation fidèle des octets du fichier. Le Read tool normalise/rend les caractères de contrôle invisibles — il ne distingue pas un espace d'un NUL d'un autre caractère invisible. Pour une question au niveau octet, c'est le mauvais outil. Et j'ai d'abord supposé "le reviewer s'est trompé" plutôt que "ma propre sortie d'outil est peut-être trompeuse".
- **Comment corrigé** : `od -c` sur les lignes concernées avant de répondre. A confirmé les NUL, infirmé ma conclusion, validé Codex. Clé reconstruite via `JSON.stringify([...])` (jamais de NUL, délimitation non ambiguë).
- **Impact** : Aucun (rattrapé avant d'envoyer la réponse). Mais c'était à un cheveu : j'allais dire à l'utilisateur que Codex avait halluciné un détail, sur un bug qui aurait crashé chaque upload en prod.
- **Lesson** : **quand un reviewer fait une affirmation au niveau octet/encodage (NUL, BOM, caractères invisibles, espaces vs tabs), vérifier avec `od -c` / `hexdump`, jamais avec le Read tool** qui normalise l'affichage. Et corollaire : avant de conclure "le reviewer s'est trompé", traiter en priorité l'hypothèse "mon outil d'observation me ment" — surtout quand le reviewer a exécuté des vérifications et pointe une ligne précise.

### 2026-05-14 — TESTING — Chemin SQL brut couvert seulement par des tests mockés → 2 bugs runtime non détectés
- **Contexte** : `acquireDocumentLineageLock` (Phase 4.3) fait un appel SQL brut Prisma : `SELECT pg_advisory_xact_lock(hashtext($key))`. Je l'ai livré avec uniquement des tests unitaires qui mockent `tx.$queryRaw` (assertion "a été appelé avec telle clé"), sans aucun test live-DB.
- **Erreur** : Deux bugs runtime DISTINCTS ont traversé toute la suite de tests verte et 2 rounds d'audit Codex sur cette même fonction : (1) clé construite avec des octets NUL → `hashtext()` rejette le NUL côté Postgres ; (2) `$queryRaw` sur une fonction retournant `void` → `P2010 Failed to deserialize column of type 'void'`. Dans les deux cas, le mock de `$queryRaw` retournait gentiment une valeur factice → tests verts, fonction cassée en réel. Le bug #2 signifiait que l'advisory lock n'a JAMAIS fonctionné — chaque upload de version / promotion aurait throw en prod.
- **Cause racine du raisonnement** : J'ai considéré qu'un test unitaire mické "couvrait" un chemin SQL brut. Mais mocker `$queryRaw`/`$executeRaw` ne teste QUE "la fonction appelle le client avec ces arguments" — ça ne teste RIEN de ce qui peut casser dans le SQL brut : compatibilité de type Postgres, désérialisation Prisma, contraintes sur les valeurs `text`, comportement pgbouncer. Le mock donne une fausse confiance proportionnelle à la complexité du SQL.
- **Comment corrigé** : Phase 5 (E2E release gate) a écrit `scripts/e2e/advisory-lock-live.ts` qui exécute la VRAIE fonction contre Postgres — il a attrapé le bug #2 immédiatement (le bug #1 avait été attrapé par Codex via `rg`/inspection binaire). Fix : `$queryRaw` → `$executeRaw`. Le script live est désormais le test de régression — un test mické ne peut structurellement pas attraper ces bugs.
- **Impact** : Bug #2 = l'advisory lock entièrement non fonctionnel sur la branche, masqué ~2 rounds d'audit. Pas de prod (jamais mergé). Coût : faux sentiment de couverture sur tout Phase 4.3.
- **Lesson** : **tout chemin SQL brut (`$queryRaw`/`$executeRaw`/`$transaction` avec du SQL littéral) DOIT avoir au moins un test live-DB** — un script reproductible ou un test d'intégration gated. Les tests mockés sur du SQL brut ne prouvent que la forme de l'appel, jamais sa validité runtime. Corollaire : quand je livre une fonction avec du SQL brut, signaler explicitement "couvert seulement par des mocks, nécessite une validation live" plutôt que de la présenter comme testée.
